Keycloak con Apple IdP
e FITP Enricher

Immagine Docker custom di Keycloak preconfezionata con due provider HiWay: login "Sign in with Apple" e arricchimento token FITP. Buildata su tutte le major Keycloak e pubblicata su GitHub Container Registry.

Repository GitHub Quickstart Troubleshooting

Cosa contiene

Due provider, un'unica immagine

Apple Identity Provider

Social IdP "Sign in with Apple" di klausbetz/apple-identity-provider-keycloak. Si configura in Identity providers → Apple.

🎫FITP Enricher

Arricchimento token HiWay/FITP di HiWay-Media/keycloak-fitp-enricher. È un Authenticator (id fitp-enricher): si aggiunge come step in Authentication → Flows.

Registry: ghcr.io/hiway-media/keycloak-apple-identity-provider-fitp-enricher

Quickstart

Avvio rapido

Run (dev)

docker run --rm -p 8080:8080 \
  -e KEYCLOAK_ADMIN=admin \
  -e KEYCLOAK_ADMIN_PASSWORD=admin \
  ghcr.io/hiway-media/keycloak-apple-identity-provider-fitp-enricher:latest \
  start-dev

Test locale con docker-compose

cp .env.example .env          # personalizza KC_IMAGE_TAG, password, ecc.
docker compose up -d
docker compose logs -f keycloak

# verifica che i provider siano registrati come SPI
docker compose --profile verify run --rm verify

Build locale

docker build \
  --build-arg KC_VERSION=26.6.1 \
  --build-arg APPLE_IDP_VERSION=1.17.0 \
  --build-arg FITP_VERSION=0.3.0 \
  -t keycloak-apple-fitp:dev .

Release (build & push automatico)

git tag v0.3.0
git push origin v0.3.0   # i workflow si attivano sul push del tag

Troubleshooting

Problemi comuni

Il FITP Enricher non compare in "Identity providers"

È corretto: il FITP Enricher non è un Identity Provider, è un Authenticator (fitp-enricher). Si configura in Authentication → Flows come step di un flow, non nella pagina Identity providers. A boot lo trovi nei log:

KC-SERVICES0047: fitp-enricher (com.hiwaymedia.keycloak.FitpEnricherAuthenticatorFactory)
is implementing the internal SPI authenticator

Lo step FITP "sparisce" dal flow dopo un redeploy

Ogni step è una riga in authentication_execution, con la colonna authenticator = l'ID del provider. Se quell'ID non corrisponde a una factory caricata, la riga resta nel DB ma lo step non viene renderizzato.

L'ID fitp-enricher è stabile in 0.2.0 → 0.2.1 → 0.3.0, quindi un bump di versione non causa il mismatch.

Diagnosi (sostituire il nome realm):

SELECT ae.authenticator, af.alias AS flow, ae.requirement, ae.flow_id
FROM authentication_execution ae
LEFT JOIN authentication_flow af ON af.id = ae.flow_id
WHERE ae.realm_id = (SELECT id FROM realm WHERE name = 'supertennix')
  AND ae.authenticator ILIKE '%fitp%';
  1. authenticator ≠ fitp-enricher (flow da una vecchia 0.1.x) → step orfano: ri-aggiungerlo dalla UI con l'ID attuale.
  2. flow NULL / flow_id inesistente → flow ricreato da un realm-import al boot. Causa più frequente del "sparisce al redeploy".
  3. Tutto coerente → cache Infinispan stale: riavvio pulito.

Prevenzione: non mischiare realm-import che sovrascrive a ogni boot con modifiche manuali in UI. Includere lo step FITP nel realm JSON versionato, oppure disabilitare l'import-on-every-boot. Backup prima con kc.sh export o Admin REST partial-export.
